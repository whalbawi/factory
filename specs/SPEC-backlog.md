# /backlog -- Backlog Tracker

## Overview

The `/backlog` skill provides persistent work-item tracking across sessions. It stores tasks
as individual markdown files with YAML frontmatter in `.factory/backlog/`, enabling both
humans and agents to create, query, update, and close work items without merge conflicts.

`/backlog` is a utility skill, not a pipeline phase. It does not appear in the `/genesis`
sequence and does not modify the main pipeline's phase state. It is invoked directly by the
user or programmatically by other skills via the auto-capture convention.

The skill exposes six subcommands: `add`, `list`, `show`, `update`, `close`, and `board`.
The default subcommand (no argument) is `list`.

---

## Contract

| Field               | Value                                                          |
|---------------------|----------------------------------------------------------------|
| **Required inputs** | Subcommand (defaults to `list` if omitted)                     |
| **Optional inputs** | Task title, task ID, flags (--type, --priority, --status,      |
|                     | --sprint), `.factory/backlog/` directory                         |
| **Outputs**         | `add`: `.factory/backlog/TASK-{NNNN}.md`                          |
|                     | `list`: formatted task list (stdout)                           |
|                     | `show`: single task details (stdout)                           |
|                     | `update`: modified `.factory/backlog/TASK-{NNNN}.md`              |
|                     | `close`: modified `.factory/backlog/TASK-{NNNN}.md`               |
|                     | `board`: `.factory/backlog/board.html` opened in browser         |
| **Failure output**  | Error message to stdout (e.g., task not found, invalid ID)     |

---

## Category

**Utility skill** -- stateless CRUD operations on a file-based task store. Not
conversational, not agentic. Each invocation performs a single operation and exits.

## Triggering

### User-Initiated

The user invokes `/backlog` directly with a subcommand:

- `/backlog add "Pin actions/create-github-app-token to commit SHA"`
- `/backlog add "Refactor deploy receipt template" --type debt --priority P2`
- `/backlog list`
- `/backlog show 3`
- `/backlog update 3 --status in_progress --sprint 2`
- `/backlog close 3`
- `/backlog board`
- `/backlog` (no argument -- runs `list`)

### Agent-Initiated (Auto-Capture)

Other skills create tasks programmatically when they defer work or produce recommendations
that are not acted on immediately. See [Auto-Capture Convention](#auto-capture-convention).

### When NOT to Use /backlog

- To plan sprints or assign work across agents -- use `/spec` Phase 2b.
- To track bugfix state -- use `/bugfix` which has its own state in `.factory/state.json`.
- To replace GitHub Issues for external collaboration -- `/backlog` is local-only.

## Storage Schema

### Directory Structure

```text
.factory/
  tasks/
    TASK-0001.md
    TASK-0002.md
    TASK-0003.md
    board.html       (generated by `board` subcommand, gitignored)
```

### Task File Format

Each task is a standalone markdown file with YAML frontmatter for structured metadata and
a markdown body for description, context, and notes.

**Example: `.factory/backlog/TASK-0001.md`**

```markdown
---
id: 1
title: Pin actions/create-github-app-token to commit SHA
type: debt
status: open
priority: P1
created_at: "2026-04-05T14:30:00Z"
created_by: user
sprint: null
---

The `create-github-app-token` action is referenced by tag (`v1`) in the release
workflow. Tags are mutable and can be force-pushed by the upstream maintainer,
which is a supply-chain risk. Pin to the specific commit SHA of the v1 release.

See: `.github/workflows/release.yml`, line 42.
```

### Frontmatter Schema

| Field        | Type   | Required | Values / Format                              |
|--------------|--------|----------|----------------------------------------------|
| `id`         | number | yes      | Sequential integer, unique across all tasks   |
| `title`      | string | yes      | Short description, max 120 characters         |
| `type`       | enum   | yes      | `bug`, `feature`, `debt`, `improvement`       |
| `status`     | enum   | yes      | `open`, `in_progress`, `done`, `closed`       |
| `priority`   | enum   | yes      | `P0`, `P1`, `P2`                              |
| `created_at` | string | yes      | ISO 8601 datetime with timezone               |
| `created_by` | string | yes      | `user` or agent/skill name (e.g., `qa`, `retro`) |
| `sprint`     | number | no       | Sprint number, `null` if unassigned           |

### Field Semantics

- **type**: `bug` = defect in existing behavior; `feature` = new capability;
  `debt` = technical debt, cleanup, refactoring; `improvement` = enhancement to
  existing capability.
- **status**: `open` = not started; `in_progress` = actively being worked on;
  `done` = work completed, pending verification; `closed` = verified complete or
  explicitly closed (won't do, duplicate, etc.).
- **priority**: `P0` = critical, blocks other work; `P1` = important, address soon;
  `P2` = nice to have, address when convenient.

### ID Assignment

The next ID is derived from the highest existing ID in `.factory/backlog/` plus one. If no
tasks exist, the first ID is 1. The filename uses zero-padded four-digit format:
`TASK-0001.md`, `TASK-0012.md`, `TASK-0100.md`.

**Collision avoidance**: When multiple agents might create tasks concurrently, each agent
reads the directory, picks `max(existing IDs) + 1`, and writes. If a collision occurs (file
already exists), increment and retry. This is acceptable because task creation is infrequent
relative to other operations.

## Process

### Skill Parameters

For the mandatory sections in [GLOBAL-REFERENCE.md](../skills/references/GLOBAL-REFERENCE.md):

- `{PHASE_NAME}` = `tasks`
- `{OUTPUT_FILES}` = `[]` (no pipeline output files; task files are the data store)

The `/backlog` skill uses the standard state tracking from GLOBAL-REFERENCE.md but with
minimal state: it records invocation timestamps but does not track phase progression since
it is a single-operation utility.

### Subcommand: `add <title>`

Create a new task.

1. Parse the title from the argument. If no title is provided, ask the user for one.
2. Parse optional flags:
   - `--type <type>` -- defaults to `feature` if omitted.
   - `--priority <priority>` -- defaults to `P1` if omitted.
3. Read `.factory/backlog/` to determine the next available ID.
4. Create `.factory/backlog/` directory if it does not exist.
5. Write `TASK-{NNNN}.md` with YAML frontmatter:
   - `id`: next available ID
   - `title`: from argument
   - `type`: from flag or default
   - `status`: `open`
   - `priority`: from flag or default
   - `created_at`: current ISO 8601 timestamp
   - `created_by`: `user` (or the calling skill name if auto-captured)
   - `sprint`: `null`
6. Body: If the user provided additional context beyond the title, include it as the
   markdown body. Otherwise, leave the body empty.
7. Confirm to the user:

```text
Created TASK-{NNNN}: {title}
  Type: {type} | Priority: {priority} | Status: open
```

### Subcommand: `list`

Display all open tasks grouped by priority. This is the default when `/backlog` is invoked
with no argument.

1. Read all `.factory/backlog/TASK-*.md` files.
2. Parse YAML frontmatter from each file.
3. Filter to tasks with `status` in (`open`, `in_progress`). Exclude `done` and `closed`.
4. Group by priority: P0 first, then P1, then P2.
5. Within each priority group, sort by ID ascending (oldest first).
6. Render as a table:

```text
## P0 -- Critical
| ID  | Title                              | Type    | Status      | Sprint |
|-----|------------------------------------|---------|-------------|--------|
| 005 | Fix auth token expiry handling      | bug     | in_progress | 3      |

## P1 -- Important
| ID  | Title                              | Type    | Status      | Sprint |
|-----|------------------------------------|---------|-------------|--------|
| 001 | Pin actions token to commit SHA     | debt    | open        | --     |
| 003 | Add retry logic to deploy health    | feature | open        | 2      |

## P2 -- Nice to Have
| ID  | Title                              | Type    | Status      | Sprint |
|-----|------------------------------------|---------|-------------|--------|
| 002 | Refactor deploy receipt template    | debt    | open        | --     |

3 open, 1 in progress, 0 done | 4 total active
```

If no active tasks exist, display:

```text
No open tasks. Use /backlog add "<title>" to create one.
```

### Subcommand: `show <id>`

Display a single task's full details.

1. Parse the ID from the argument. Accept bare numbers (`3`) or padded (`003`).
2. Read `.factory/backlog/TASK-{NNNN}.md`.
3. If the file does not exist, display an error: `Task {id} not found.`
4. Display the full frontmatter as a formatted summary, followed by the markdown body:

```text
TASK-0003: Add retry logic to deploy health check
  Type: feature | Priority: P1 | Status: open
  Created: 2026-04-05 by user | Sprint: 2

  The deploy health check fails intermittently on cold starts because the
  first request times out. Add a 3-retry loop with exponential backoff.
```

### Subcommand: `update <id>`

Update one or more fields on an existing task.

1. Parse the ID from the argument.
2. Parse flags: `--status <status>`, `--priority <priority>`, `--type <type>`,
   `--sprint <number>`, `--title <title>`.
3. Read the task file. If it does not exist, display an error.
4. Update the specified frontmatter fields. Do not modify fields that were not specified.
5. Write the updated file back.
6. Confirm:

```text
Updated TASK-{NNNN}:
  status: open -> in_progress
  sprint: -- -> 2
```

### Subcommand: `close <id>`

Close a task. Shorthand for `update <id> --status closed`.

1. Parse the ID from the argument.
2. Read the task file. If it does not exist, display an error.
3. Set `status` to `closed`.
4. Write the updated file back.
5. Confirm:

```text
Closed TASK-{NNNN}: {title}
```

### Subcommand: `board`

Render an HTML kanban board and open it in the browser.

1. Read all `.factory/backlog/TASK-*.md` files.
2. Parse YAML frontmatter from each file.
3. Group tasks into columns by status: `open`, `in_progress`, `done`, `closed`.
4. Within each column, sort by priority (P0 first) then by ID ascending.
5. Generate an HTML file with:
   - Four columns: Open, In Progress, Done, Closed.
   - Each card shows: ID, title, type badge, priority badge.
   - Color coding: P0 = red border, P1 = amber border, P2 = grey border.
   - Type badges: bug = red, feature = blue, debt = yellow, improvement = green.
   - Inline CSS (no external dependencies).
   - Responsive layout that works at common viewport widths.
6. Write to `.factory/backlog/board.html`.
7. Open in browser: `open .factory/backlog/board.html`.

The board is a snapshot, not a live view. Re-run `/backlog board` to refresh.

## Auto-Capture Convention

Skills may create tasks when they encounter deferred work or produce recommendations that
are not acted on immediately. This is opt-in -- skills are not required to create tasks.

### Convention

When a skill identifies work that should be tracked but is out of scope for the current
operation, it creates a task by writing a `TASK-{NNNN}.md` file directly (following the
same ID assignment logic as the `add` subcommand).

**When to auto-capture:**

- `/qa` finds a minor, non-blocking issue that does not fail the gate.
- `/retro` produces an action item or process improvement recommendation.
- `/security` identifies a low-severity finding that does not block deploy.
- `/build` defers a known optimization or cleanup to a future sprint.
- `/deploy` notes a configuration improvement for next deployment.

**When NOT to auto-capture:**

- Gate-blocking findings. These halt the pipeline -- they are not deferred, they are
  blocking.
- Items already tracked. Check `.factory/backlog/` before creating a duplicate.
- Vague suggestions. Tasks must have a concrete title and enough context to act on.

### Auto-Captured Task Format

Auto-captured tasks follow the same schema as user-created tasks, with these conventions:

- `created_by`: the skill name (e.g., `qa`, `retro`, `security`).
- `type`: matches the nature of the work (e.g., `bug` for a QA finding, `improvement` for
  a retro action item, `debt` for a deferred cleanup).
- `priority`: set by the capturing skill based on severity/impact.
- Body: includes context from the capturing skill -- the finding, recommendation, or
  rationale. References the source report (e.g., "See QA-REPORT.md, finding #3").

### GLOBAL-REFERENCE.md Integration

The auto-capture convention is documented as an optional section in GLOBAL-REFERENCE.md.
Skills that opt in reference this section. The section provides the convention but does not
mandate it -- each skill's `SKILL.md` decides whether to participate.

**Proposed GLOBAL-REFERENCE.md section:**

```markdown
## Task Auto-Capture [OPTIONAL]

When this skill identifies work that is out of scope for the current operation but
should be tracked for future action, create a task file in `.factory/backlog/`:

1. Read `.factory/backlog/` to determine the next available ID (max existing ID + 1).
2. Create `.factory/backlog/TASK-{NNNN}.md` with frontmatter:
   - `created_by`: this skill's name
   - `type`: matches the nature of the deferred work
   - `priority`: based on severity/impact assessment
   - `status`: `open`
3. Include context in the body: what was found, why it was deferred, and a reference
   to the source report.

Only create tasks for concrete, actionable items. Do not create tasks for gate-blocking
findings (those halt the pipeline) or vague suggestions.
```

## Sprint Integration

### How /spec Reads the Backlog

During Phase 2b (sprint planning), `/spec` reads `.factory/backlog/` to incorporate
existing work items into the sprint plan:

1. `/spec` reads all `TASK-*.md` files with `status: open` or `status: in_progress`.
2. Tasks with matching `type` are considered for inclusion:
   - `bug` and `improvement` tasks are candidates for any sprint.
   - `feature` tasks are candidates when they align with the sprint's theme.
   - `debt` tasks are candidates when the sprint has capacity.
3. `/spec` presents the backlog to the user during sprint planning:

```text
Open backlog items that may fit this sprint:

P0:
  - TASK-0005: Fix auth token expiry handling (bug)

P1:
  - TASK-0001: Pin actions token to commit SHA (debt)
  - TASK-0003: Add retry logic to deploy health check (feature)

P2:
  - TASK-0002: Refactor deploy receipt template (debt)

Include any of these in Sprint 2? [select by ID / skip]
```

4. For tasks the user selects, `/spec` updates the task's `sprint` field to the current
   sprint number via the same frontmatter-update mechanism as the `update` subcommand.
5. Selected tasks become work items in the sprint plan alongside spec-derived tasks.

### Backlog-Driven Sprint Awareness

Tasks that already have a `sprint` field set (from a previous planning session or manual
assignment) are shown as already assigned:

```text
Already assigned to Sprint 2:
  - TASK-0003: Add retry logic to deploy health check (feature, P1)
```

This prevents double-assignment and gives `/spec` visibility into pre-committed work.

## Acceptance Criteria

| Criterion | How to verify |
|-----------|---------------|
| `add` creates a valid task file | Run `/backlog add "Test task"`, verify `.factory/backlog/TASK-0001.md` exists with correct frontmatter |
| IDs are sequential and unique | Create 3 tasks, verify IDs are 1, 2, 3. Delete task 2, create another, verify ID is 4 (not 2) |
| `list` groups by priority | Create tasks at P0, P1, P2. Run `/backlog list`, verify P0 appears first |
| `list` filters out closed/done | Close a task, run `/backlog list`, verify it does not appear |
| `show` displays full details | Run `/backlog show 1`, verify frontmatter and body are shown |
| `update` modifies only specified fields | Run `/backlog update 1 --priority P0`, verify only priority changed |
| `close` sets status to closed | Run `/backlog close 1`, verify frontmatter shows `status: closed` |
| `board` generates valid HTML | Run `/backlog board`, verify `.factory/backlog/board.html` exists and opens |
| Auto-capture creates valid tasks | Simulate a skill creating a task with `created_by: qa`, verify format |
| Sprint field is nullable | Create a task, verify `sprint: null`. Update with `--sprint 2`, verify `sprint: 2` |
| ID collision recovery | Simulate two agents creating tasks concurrently, verify no data loss |
| Empty backlog handled gracefully | Run `/backlog list` with no task files, verify friendly message |
| `/backlog` with no argument runs `list` | Invoke `/backlog` with no argument, verify list output |

## Decision Log

| Decision | Rationale | Reversible |
|----------|-----------|------------|
| Directory-based storage (one file per task) | Avoids merge conflicts when multiple agents create tasks concurrently. Each file is independently readable and writable. | Yes -- could migrate to a single file later |
| Sequential integer IDs | Simple, human-friendly, easy to reference in conversation. UUIDs are overkill for a local task store. | Yes |
| YAML frontmatter + markdown body | Consistent with Factory's existing patterns (BUGFIX-{id}.md, SPEC.md). Structured metadata is queryable; free-form body handles context. | No -- changing format requires migration |
| Default subcommand is `list` | Most common operation. Typing `/backlog` should immediately show the backlog. | Yes |
| Auto-capture is opt-in | Prevents noise from skills that don't produce deferred items. Each skill decides whether to participate. | Yes -- could make mandatory later |
| `board.html` is generated, not persisted | The board is a snapshot view, not a source of truth. Regenerating is cheap. File should be gitignored. | Yes |
| Sprint field is optional and nullable | Not all tasks belong to a sprint. Unassigned tasks sit in the backlog until planned. | Yes |
| Status enum includes `done` and `closed` | `done` = work finished, pending verification. `closed` = verified or explicitly closed. This mirrors common issue-tracker semantics. | Yes |
| P0/P1/P2 priority scale | Consistent with `/bugfix` severity levels. Three levels are enough for a local backlog. | Yes |
| No state.json namespace | Unlike `/bugfix`, `/backlog` does not need pipeline state. Task files ARE the state. The skill updates `.factory/state.json` per GLOBAL-REFERENCE.md but only for invocation tracking. | Yes |
